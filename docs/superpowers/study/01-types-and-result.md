# 01 — Types & Result Pattern

**Source:** `packages/flint/src/types.ts`

## Role

`Role` is a string union of four values: `system`, `user`, `assistant`, and `tool`. The first three map directly to the roles recognized by every major provider API. `tool` is kept as a distinct role rather than a subtype of `user` because providers handle tool results structurally differently from ordinary user messages. Anthropic coalesces tool results into a user turn on the wire but requires the `tool_result` block type to be embedded inside it; OpenAI uses `role: "tool"` with a `tool_call_id` field at the top level of the message object. By treating `tool` as its own role in Flint's type system, adapters can branch on `message.role === 'tool'` and emit the correct wire format without inspecting content shape or relying on out-of-band flags.

## ContentPart

`ContentPart` is a discriminated union with three variants keyed on `type`. The `text` variant carries a `text: string` and is the common case for all text content. The `image` variant carries a `url: string`; the provider fetches the image at inference time, so the payload is minimal. The `image_b64` variant carries `data: string` (raw base64-encoded bytes) and `mediaType: string` (e.g. `"image/jpeg"`, `"image/png"`); this variant is used when the image is not publicly reachable over the internet and must be sent inline. Only `user` messages accept `ContentPart[]` as their content value; every other role uses `string`. This constraint is encoded directly in the `Message` union rather than left to convention.

## ToolCall

`ToolCall` has three fields. `id: string` is generated by the provider when it decides to invoke a tool and must be echoed back verbatim in the corresponding tool-role message so the provider can correlate result to call; losing or mutating this id causes the provider to reject the conversation. `name: string` is the tool's registered name, used by `execute` to look up the correct `Tool` definition. `arguments: unknown` is typed as `unknown` rather than `Record<string, unknown>` deliberately: the Standard Schema validator attached to the `Tool` definition is the authority on argument shape, and narrowing happens inside `execute` after validation. Typing arguments as `Record<string, unknown>` would imply a structure that isn't guaranteed before validation runs.

## Message

`Message` is a four-branch discriminated union where `role` is the discriminant. The `system` branch accepts only `content: string` — no multi-modal content, no tool data. The `user` branch accepts `content: string | ContentPart[]`, supporting plain text and multi-modal inputs. The `assistant` branch carries `content: string` for the model's text output and an optional `toolCalls?: ToolCall[]`; both can be present simultaneously in a single turn, which happens when a model emits a reasoning preamble before calling tools. The `tool` branch carries `content: string` (the serialized result of executing the tool, typically JSON) and `toolCallId: string` linking it to the originating `ToolCall.id`. The discriminated union design lets TypeScript narrow to the precise type at each branch without casting, making adapter code and agent loop logic both safe and self-documenting.

## ToolPermissions

`ToolPermissions` is a flat object type with five optional boolean and array fields, all optional so tools with no declared permissions are treated as unrestricted by default. `destructive?: boolean` is an advisory flag that UI layers and approval gates can inspect to warn users before execution proceeds. `scopes?: string[]` is consumed by `permissionedTools` to filter which tools are eligible in a given execution context. `network?: boolean` and `filesystem?: boolean` are advisory flags that describe the tool's external access patterns without enforcing anything at the runtime level. `requireApproval?: boolean` is set automatically by the `requireApproval()` wrapper rather than by tool authors directly; when `true`, the agent runtime must pause and request human confirmation before calling `execute`.

## Tool\<Input, Output\>

`Tool` takes two generics that flow through the definition in a single chain: the `input` field holds a `StandardSchemaV1<unknown, Input>` validator, and `handler` is typed as `(input: Input) => Promise<Output> | Output`. This means the TypeScript type of validated arguments and the TypeScript type of the handler parameter are guaranteed to match by construction. The `jsonSchema?: Record<string, unknown>` field is a manual override for the JSON Schema sent to the provider. When absent, the adapter falls back to `{ type: 'object' }` as a catch-all; when present, it replaces whatever the Standard Schema library would derive. This escape hatch exists because complex validators — particularly those with union branches or recursive types — can produce JSON Schema that specific providers reject. `timeout?: number` specifies the maximum execution time in milliseconds; enforcement is handled inside `execute` and is not the tool author's responsibility.

## Result\<T, E\>

`Result<T, E>` is a two-branch discriminated union: `{ ok: true; value: T }` or `{ ok: false; error: E }`, with `E` defaulting to `Error`. Every async function in Flint's primitives layer returns `Promise<Result<T>>` rather than throwing. The motivation is straightforward: callers handle both branches with a plain `if (!res.ok)` check, TypeScript exhaustively narrows `value` in the success branch and `error` in the failure branch, and there is no try/catch nesting to manage. The only places Flint throws unconditionally are programmer-error conditions — passing bad arguments to `budget()`, omitting required constructor options — where recovery is not meaningful. All recoverable runtime failures, including network errors, provider rejections, tool timeouts, and validation mismatches, are returned as `{ ok: false; error }`.

## Usage

`Usage` records token counts for a single inference call: `input: number` for prompt tokens, `output: number` for completion tokens, and `cached?: number` for prompt-cache hits. `cached` is optional because most providers do not report cache hit counts; Anthropic is currently the only provider that does. The budget system accumulates all three fields across calls to compute total spend and enforce token limits.

## StopReason

`StopReason` is a string union of four values. `end` means the model finished generating naturally. `tool_call` means the model stopped because it wants one or more tools executed — this is the branch the agent loop checks to decide whether to continue iterating. `max_tokens` means the response was cut off by the context or completion token limit. `stop_sequence` means a caller-supplied stop string was encountered. The `call` primitive also branches on `tool_call` to skip text-content parsing: when the model stopped to use a tool there is no assistant text to extract, so attempting to parse it would be incorrect.

## StreamChunk

`StreamChunk` is a four-variant union emitted in a defined order during a streaming response. `{ type: 'text'; delta: string }` carries incremental text; callers concatenate deltas to reconstruct the full response. `{ type: 'tool_call'; call: ToolCall }` is emitted once per tool call after its arguments are fully assembled — tool call arguments are not streamed character-by-character, so this chunk arrives as a complete unit. `{ type: 'usage'; usage: Usage; cost?: number }` carries final token counts and an optional cost estimate at the end of the stream. `{ type: 'end'; reason: StopReason }` carries the stop reason as the final chunk. The `stream` primitive re-yields all four variants in sequence; callers can handle whichever types they care about and discard the rest with a type-narrowed switch.

## Logger

`Logger` declares four methods — `debug`, `info`, `warn`, and `error` — each typed as an optional method taking a `msg: string` and an optional `meta?: unknown`. All four are optional, so callers can supply a partial logger that only implements the methods they care about. The interface is structurally compatible with the global `console` object, meaning `console` can be passed directly anywhere a `Logger` is accepted. Flint's core primitives do not call `Logger` methods themselves in the current codebase; the interface exists as a shared contract for adapters and application code to wire up their own logging infrastructure without taking a dependency on a specific logging library.
